[1:00 - ARCHITECTURE]
(show assets/architecture_diagram.png)
Let's take a look at the system architecture. (pause)
First, every query goes through our Security Checkpoint node to redact personal medical data and block injection attempts. (pause)
From there, the sanitized request goes to the HealthSync Orchestrator. (pause)
The Orchestrator delegates to three specialized LlmAgents: the Vitals Tracker Agent, the Symptom Checker Agent, and the Scheduler Agent. (pause)
These agents interact with a local FastMCP Server to commit data to safety logs or schedule calendar slots. (pause)
If scheduling a doctor's visit is requested, the system automatically redirects to our Appointment Approval Node for a human-in-the-loop validation step before finalizing.

[2:00 - LIVE DEMO]
(switch to playground at localhost:18081)
Let's see it in action. (pause)
First, we'll log a vital sign. If I type "Can you log my blood sugar as 110 mg/dL?", the orchestrator routes this to the Vitals Tracker sub-agent. The agent logs the sugar value via the MCP tool and confirms it immediately. If we follow up with "Show me my blood sugar history", it queries the history tool and formats the results in a tidy list. (pause)
Next, let's test medication safety. If I ask "I'm taking Aspirin and Ibuprofen. Are there any interactions?", the symptom agent is triggered. It queries the local contraindication tool, identifying a bleeding risk warning, and alerts the patient to consult a pharmacist. (pause)
Lastly, let's schedule a checkup. I'll type: "I need to schedule a checkup with Dr. Smith tomorrow at 10 AM". The scheduler agent writes a pending booking to the state, routes to the approval node, and triggers a RequestInput prompt. The UI shows a pause, asking us: "Please confirm: Do you want to schedule this appointment?". Once I enter 'yes', the workflow resumes, commits the appointment, and shows booking success.
